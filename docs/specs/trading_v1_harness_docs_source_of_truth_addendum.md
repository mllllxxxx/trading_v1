# trading_v1 — Harness Docs Demotion and Source-of-Truth Boundary Addendum

Status: Addendum for the first post-governance PR  
Audience: coding agents, maintainers, reviewers  
Scope: documentation hierarchy, source-of-truth boundaries, RAG/prompt indexing boundaries, and development-process docs

This addendum should be applied after the initial source-of-truth governance PR. It closes a specific governance gap: the repository still contains harness-oriented documentation and agent instructions that can be useful for development process, but must not be treated as trading product policy, runtime configuration, LLM trading context, or rulebook source of truth.

The goal is not to delete harness assets. The goal is to demote and isolate them so that future coding agents do not confuse generic repository-harness instructions with the trading system's canonical policy.

---

## 1. Problem statement

The repository contains a real trading application under `trading/`, but it also contains repository-harness documents, templates, and instructions. These harness files can mislead humans or coding agents if they appear to be canonical product architecture or trading policy.

The main risk is not that harness docs directly place trades. The main risk is source-of-truth drift during development:

```text
- An AI coding agent reads generic harness docs first.
- It thinks the repository is a generic agent-ready workspace, not a trading system.
- It adds trading policy to README, AGENTS.md, prompts, scheduler, or validator.
- The rulebook/source governance model is bypassed.
- Runtime behavior begins to diverge from the canonical rulebook and schemas.
```

This must be prevented before deeper refactors such as rulebook retrieval, TradeDecisionTicket enforcement, critic agent, verifier, risk compiler, forex adapter, or live-readiness work.

## 3. Definitions

### 3.1 Trading source-of-truth files

These files are allowed to define trading behavior, product boundaries, risk constraints, LLM role, runtime contracts, or policy source locations.

Examples:

```text
docs/product/*.md
docs/architecture/SOURCE_OF_TRUTH_MAP.md
docs/architecture/DECISION_FLOW.md
docs/architecture/RUNTIME_CONTEXT_BOUNDARIES.md
docs/architecture/RAG_INDEXING_POLICY.md
docs/decisions/ADR-*.md
trading/rulebook/source/**/*.yaml
trading/rulebook/source/**/*.json
trading/schemas/**/*.py
trading/schemas/**/*.json
trading/config/**/*.yaml
trading/config/**/*.json
```

### 3.2 Generated artifacts

These files are derived from canonical source and should not be edited manually.

Examples:

```text
trading/rulebook/compiled/*.json
trading/rulebook/rendered/*.md
trading/auto/skills.json
```

Generated files must contain a clear header:

```text
DO NOT EDIT - generated from canonical source-of-truth files.
```

### 3.3 Development-process docs

These files can describe how agents should work, how features are proposed, how checklists are used, or how the repository harness operates. They must not define trading policy.

Examples after demotion:

```text
docs/harness/HARNESS.md
docs/harness/FEATURE_INTAKE.md
docs/harness/TEST_MATRIX.md
docs/harness/TOOL_REGISTRY.md
docs/harness/CONTEXT_RULES.md
docs/harness/GENERIC_ARCHITECTURE_TEMPLATE.md
```

### 3.4 Runtime trading context

Runtime trading context is the data passed to LLM Trader, critic, verifier, risk compiler, or execution router.

Allowed runtime context sources:

```text
- MarketDossier
- RetrievedRuleContext from trading/rulebook/rendered and trading/rulebook/compiled
- Product/risk/autonomy docs explicitly allowed by RAG_INDEXING_POLICY
- Portfolio/account snapshot
- Broker/exchange state
- Journal summaries generated by the journal module
- Event/macro data from approved data providers
```

Disallowed runtime context sources:

```text
- docs/harness/**
- AGENTS.md
- README.md
- generic templates
- repository-harness scripts
- old architecture docs that are not explicitly marked canonical
```

### 5.1 Root README.md

Root README should identify the repository as the trading system, not as a generic repository harness.

Required content:

```md
# trading_v1

This repository contains an LLM-governed trading system.

The active trading application lives in:

- `trading/`

The active source-of-truth and architecture specifications live in:

- `trading/schemas/`
- `trading/config/`

Harness documents are retained only as development-process guidance for coding agents.
They are not trading policy, runtime configuration, LLM trading context, broker policy, or rulebook source of truth.
```

Root README may link to harness docs, but the link text must say they are process docs only.

Root README must not contain canonical values for:

```text
- risk per trade
- leverage
- max positions
- confidence thresholds
- R:R thresholds
- broker execution behavior
- LLM output schema
- playbook logic
- symbol universe
```

- prompt policy

## 14. Summary

Harness docs are allowed to exist, but they must be clearly demoted.

Correct relationship:

```text
Harness docs
  -> development process guidance only

Governance/spec/product/architecture/rulebook/schema/config
  -> trading source of truth

Runtime modules
  -> read compiled/config/schema/rulebook sources only

LLM Trader
  -> receives market dossier + retrieved rulebook/product context only

Verifier/risk compiler
  -> enforce compiled hard rules and risk profiles
```

This prevents future AI-assisted development from accidentally turning generic harness documentation into trading policy or LLM runtime context.
